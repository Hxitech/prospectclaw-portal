Human Guide

按真实使用顺序来，不按概念先后排

你可以把这部分理解成“用户第一次装好之后，应该怎么把 ProspectClaw 用起来”。

推荐顺序

安装 ProspectClaw → 跑完初始设置 → 先配置钳子助手 → 再配置 OpenClaw → 最后接渠道和排障

先把钳子助手配好，是因为它是最稳定的“第二双眼睛”。后面模型不通、Gateway 崩了、Agent 不回话、飞书不响应，你都可以先让钳子助手帮你判断从哪一层开始查。

01
安装与第一次启动

先把应用装好，再跑一遍初始设置

ProspectClaw 的桌面包安装完成后，第一次启动会进入 `初始设置`。这里不是装饰页，而是整个系统真正的起跑线。

  • macOS 版首次打开如果被拦截，先把应用拖进 `应用程序`，必要时去系统设置里点“仍要打开”。
  • Windows 遇到安装 OpenClaw 报错，优先检查管理员权限、npm 缓存和 Git HTTPS 配置。
  • Linux 如果没有桌面环境，建议直接走 Web 版部署路径，通过浏览器管理。
过关标准： `初始设置` 里的 Node.js、Git、OpenClaw、配置文件都已经通过，知道自己下一步该去哪里。

桌面端

适合 Mac / Windows 用户，下载后直接安装，首次启动自动进入初始设置。

Web 版

适合 Linux / 服务器场景。默认通过 `:1420` 提供面板，通过 `:18789` 管理 Gateway。

02
先把修理工叫醒

先配置钳子助手

很多人会直接去模型、Agent、聊天，然后出问题时又回头配钳子助手。更稳的做法是反过来：先把钳子助手自己的模型配好，让它先能工作。

  • 钳子助手是独立于 OpenClaw Agent 的内置 AI 助手。
  • 它擅长解释报错、分析日志、推荐下一步、调用内置卡片做排障。
  • 钳子助手欢迎页自带卡片，像 `检查系统环境`、`诊断 Gateway`、`一键排障`、`Skills 管理` 这些都可以直接点。
过关标准： 钳子助手自己已经能对话。到这一步，你已经有一个随时能问的“运维同事”。
03
配置 OpenClaw 主体

配置模型，然后启动 Gateway

`模型配置` 页面决定 OpenClaw 用什么模型工作，`服务管理` 页面决定 Gateway 是否正常跑起来。这两个步骤加起来，才算把 OpenClaw 主体点亮。

  • 至少添加 1 个主模型，建议再准备 1 到 2 个备选模型。
  • 模型测试通过后保存，必要时重启 Gateway 让配置生效。
  • `服务管理` 里看 Gateway 是否绿色运行，出问题优先看这里和 `日志查看`。
过关标准： 主模型可用，Gateway 能正常启动，服务状态不是红的。
04
创建和验证 Agent

创建 Agent，再去实时聊天验收

`Agent 管理` 负责创建和维护你的角色实例，`实时聊天` 才是最终验收场。不要只看列表里有没有 Agent，要看它是不是真的能回复。

  • 第一次上手建议先做单一职责 Agent，不要一开始就堆太多设定。
  • 如果要深度改规则、工作区或风格，可以用聊天式配置或高级向导。
  • 最终一定要去 `实时聊天` 发第一条消息，确认 Gateway、模型、Agent、权限链路都通了。
过关标准： Agent 在实时聊天里能稳定回答，且你知道它绑定的是哪个模型。
05
接消息渠道

先明白规律，再接飞书和 Telegram

`消息渠道` 页面统一管理外部接入。无论接什么平台，有两个前提都一样：Gateway 要先运行，保存配置后通常都需要做平台侧的发布或配对审批。

  • 飞书 / Lark：创建企业自建应用，开启机器人能力，拿到 App ID / App Secret，事件订阅建议用长连接（WebSocket），保存后做配对审批。
  • Telegram：通过 `@BotFather` 创建 Bot，拿到 Bot Token，再准备允许的用户 ID，填到渠道配置里保存即可。
  • 保存后不生效，优先检查 Gateway 是否在运行、平台侧应用是否已发布、机器人是否已被添加到群或工作台中。
过关标准： 机器人已经能在对应平台收到并回复消息，或者你至少已经完成凭证校验和配对。

飞书 / Lark

需要 App ID、App Secret，支持内置插件和官方插件两种模式。发布应用版本后，机器人对其他人才可见。

Telegram

需要 Bot Token 和允许的用户 ID。最轻量，适合最快跑通第一条外部渠道。

日常会高频用到的页面

  • `钳子助手`：问路、排障、卡片触发。
  • `服务管理`：启动 / 停止 / 重启 Gateway，做配置备份。
  • `日志查看`：看 Gateway 日志、错误日志、守护进程日志。
  • `记忆文件`：按 Agent 查看工作记忆、归档和核心文件。
  • `关于`：看推荐稳定版，做版本切换或回退。

一套够用的排障顺序

  • 先让钳子助手帮你缩小范围。
  • 再看 `初始设置` 和 `模型配置`。
  • 然后看 `服务管理` 和 `日志查看`。
  • 最后再看具体 Agent、Skills 或消息渠道。
  • 除非已经确认彻底损坏，否则不要一上来就重装。
Quick FAQ

常见问题，先记这几条

🍎

macOS 提示“已损坏”

通常是未签名应用被 Gatekeeper 拦截。先把应用拖进 `应用程序`，必要时再去系统设置放行。

🧱

Node.js 检测不到

Windows 上常见于 PATH 没刷新；macOS 上常见于从 Finder 启动时路径不完整。先重启应用，不行再让钳子助手做环境检查。

🚦

Gateway 启动失败

优先看端口占用、配置文件格式、模型配置和认证模式。最省事的入口通常是钳子助手的 `诊断 Gateway` 或 `一键排障`。

📡

渠道保存后不生效

先确认 Gateway 正在运行，再检查平台侧是否发布应用、机器人是否被添加进群或工作台、是否完成配对审批。

# ProspectClaw 使用知识库（面向 AI）

## 1. 产品定位
- ProspectClaw 是一个本地化的 OpenClaw 管理面板。
- 在当前产品叙事中，它会展示投资工作台，但同时保留完整的 OpenClaw 管理能力。
- 当用户问“ProspectClaw 怎么用”，优先回答面板的实际操作路径，而不是先解释 OpenClaw 的抽象概念。

## 2. 推荐上手顺序
1. 安装并启动 ProspectClaw。
2. 跑完“初始设置”，确认 Node.js、Git、OpenClaw、配置文件就绪。
3. 先配置“钳子助手”自己的模型，让钳子助手先能工作。
4. 进入“模型配置”，至少添加一个可用主模型。
5. 进入“服务管理”，启动 Gateway。
6. 进入“Agent 管理”，创建或调整 Agent。
7. 进入“实时聊天”，验证 Agent 是否真正能回复。
8. 如需外部接入，进入“消息渠道”配置飞书、Telegram、Discord、钉钉、QQ 等。

## 3. 核心页面职责
- 初始设置：环境检查、OpenClaw 初始化入口。
- 钳子助手：独立 AI 助手，可问答、排障、触发内置卡片。
- 模型配置：服务商、模型、主模型、备选模型配置。
- 服务管理：Gateway 启停、版本信息、配置备份。
- Agent 管理：创建、编辑、备份、配置 Agent。
- 实时聊天：验证 Gateway + 模型 + Agent 链路是否可用。
- 消息渠道：飞书、Telegram、Discord、钉钉、QQ 接入。
- 日志查看：Gateway 日志、错误日志、守护日志、审计日志。
- 记忆文件：按 Agent 查看和编辑工作记忆、归档、核心文件。
- 关于：版本、推荐稳定版、切换 / 回退 OpenClaw 版本。

## 4. 关于钳子助手
- 钳子助手是 ProspectClaw 内置的独立 AI 助手，不等于用户的 OpenClaw Agent。
- 推荐在用户刚装好产品时，先配置钳子助手自己的模型。
- 钳子助手常用于：
  - 解释安装或配置报错
  - 分析日志
  - 推荐下一步操作
  - 触发内置卡片做排障
- 钳子助手当前常见内置卡片包括：
  - 检查 OpenClaw 配置
  - 诊断 Gateway
  - 浏览配置目录
  - 检查系统环境
  - 分析错误日志
  - 一键排障
  - Skills 管理
  - 提交 Bug 报告
  - PR 助手
  - 调试现有 Agent

## 5. 模型与 Gateway
- 用户至少需要配置一个可用主模型，建议有备选模型。
- 模型测试通过后，保存配置；必要时重启 Gateway 生效。
- Gateway 是 OpenClaw 的核心服务入口，聊天、渠道接入、很多 Agent 行为都依赖它。
- 当用户反馈“实时聊天无响应”时，优先排查：
  1. Gateway 是否运行
  2. 端口 18789 是否正常监听
  3. 模型是否可用
  4. Agent 是否绑定模型
  5. 认证或权限是否拦截

## 6. Feishu / Lark 渠道接入要点
- 在飞书开放平台创建企业自建应用。
- 为应用开启机器人能力。
- 获取 App ID 和 App Secret。
- 建议使用长连接（WebSocket）做事件订阅。
- 在消息渠道页面填写：
  - App ID
  - App Secret
  - domain（feishu 或 lark）
  - pluginVersion（builtin 或 official）
- 保存后需注意：
  - Gateway 必须运行
  - 可能需要完成配对审批
  - 飞书应用需要发布版本后，机器人对其他人可见
- 如果机器人“配置保存成功但在飞书里找不到”，优先检查发布状态和可见范围。

## 7. Telegram 渠道接入要点
- 通过 @BotFather 创建 Telegram Bot。
- 获取 Bot Token。
- 通过 @userinfobot 等方式获取自己的 Telegram 用户 ID。
- 在消息渠道页面填写：
  - botToken
  - allowedUsers
- Telegram 是最轻量、最适合先跑通的外部渠道之一。

## 8. 常见问题回答策略
- macOS 提示“已损坏”：
  - 先确认应用已拖入应用程序目录。
  - 再指导用户到系统设置中选择“仍要打开”。
- Node.js 检测不到：
  - Windows 先提醒重启应用刷新 PATH。
  - macOS 若从 Finder 启动可尝试从终端打开应用。
- Gateway 启动失败：
  - 排查端口占用、配置文件损坏、模型配置异常、认证模式错误。
  - 推荐用户先用钳子助手“诊断 Gateway”或“一键排障”。
- 模型连接超时：
  - 检查 API Key、Base URL、网络代理、服务商侧可用性。
- 渠道保存后不生效：
  - 先确认 Gateway 在运行
  - 再确认平台侧应用已发布
  - 再确认机器人已加入群或工作台
  - 再确认配对审批已完成

## 9. 回答原则
- 当用户问“ProspectClaw 怎么用”时，优先讲页面路径和实际动作。
- 当用户问“为什么不工作”时，先缩小层级：环境 / 模型 / Gateway / Agent / Skills / 渠道。
- 当用户是非投资用户时，不要默认把投资工作台作为主路径。
- 当用户只是要把 OpenClaw 跑起来时，优先推荐：
  初始设置 → 钳子助手 → 模型配置 → 服务管理 → Agent 管理 → 实时聊天 → 消息渠道
