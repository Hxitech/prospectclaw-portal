01
安装与第一次启动

先把应用装好，再跑一遍初始设置

ProspectClaw 的桌面包安装完成后，第一次启动会进入 `初始设置`。这里不是装饰页，而是整个系统真正的起跑线。

  • macOS 版首次打开如果被拦截，先把应用拖进 `应用程序`，必要时去系统设置里点"仍要打开"。
  • Windows 遇到安装 OpenClaw 报错，优先检查管理员权限、npm 缓存和 Git HTTPS 配置。
  • Linux 如果没有桌面环境，建议直接走 Web 版部署路径，通过浏览器管理。
过关标准： `初始设置` 里的 Node.js、Git、OpenClaw、配置文件都已经通过，知道自己下一步该去哪里。

桌面端

适合 Mac / Windows 用户，下载后直接安装，首次启动自动进入初始设置。

Web 版

适合 Linux / 服务器场景。默认通过 `:1420` 提供面板，通过 `:18789` 管理 Gateway。

02
先把钳子助手配好

先配置钳子助手，不要等出事了才想起它

钳子助手和你的 OpenClaw Agent 不是一回事。它是 ProspectClaw 内置的独立 AI 助手，适合做安装说明、配置建议、日志分析和排障。换句话说，它更像你的"随身教练"和"急诊科医生"。

  • 先去 `钳子助手` 页面，把它自己的模型配置好。
  • 这样就算你的 OpenClaw Agent 还没养活，你也已经有一个能问问题、能看报错、能给修复建议的帮手。
  • 后面如果模型配崩了、Gateway 抽风了、Skill 缺依赖了，都可以先把报错甩给钳子助手分析。
过关标准： 钳子助手自己已经能正常对话。到这一步，你的"修理工"已经上岗了。
Assistant Cards

钳子助手欢迎页自带一组内置卡片

这也是它和普通聊天框最不一样的地方。很多常见操作你不用从零描述，直接点卡片，钳子助手就会按预设流程开始检查、分析或排障。对新手来说，这比"我该怎么提问才对"省脑子得多。

🔧

检查 OpenClaw 配置

读取并分析 `openclaw.json`，帮你看服务商、Gateway 和常见配置错误。

适合：刚装完、刚改完配置、怀疑写错字段时
🏥

诊断 Gateway

检查进程、端口和日志，判断 Gateway 是没启动、没监听，还是启动了但状态不对。

适合：实时聊天连不上、端口异常、网关抽风时
📂

浏览配置目录

把 `.openclaw` 目录结构摊开给你看，告诉你配置、日志、Agent、备份分别在哪里。

适合：刚上手时认路，或想知道文件该去哪找时
💻

检查系统环境

检查 Node.js、npm、OpenClaw CLI 和端口状态，帮你确认底层环境够不够跑。

适合：安装卡住、升级后异常、怀疑环境不完整时
📋

分析错误日志

从日志里抓 `ERROR`、`WARN`、`fail` 这些关键词，帮你缩小问题范围。

适合：看得见报错，但一时看不懂时
🔨

一键排障

把配置、模型同步、Gateway、WebSocket、Node.js 这些常见环节串起来做一轮体检。

适合：虾突然蔫了、不知道先查哪一层时
🧰

Skills 管理

查看哪些 Skills 可用、哪些缺依赖，还能引导你安装缺失依赖。

适合：Agent 会聊天但不会干活时
🐛

提交 Bug 报告

帮你整理环境信息、复现步骤和日志，生成能直接拿去提 Issue 的内容。

适合：已经确认是产品问题，准备反馈时
🔀

PR 助手

定位问题、整理修复思路，并帮你生成更像样的 PR 描述和提交流程。

适合：你准备自己修，或者想让它先帮你理清方案时
🕵️

调试现有 Agent

围绕某只现有 Agent 的文件、规则和运行状态做分析型调试。

适合：明明只有某一只虾不对劲时
用法建议： 不确定该点哪张卡时，优先从 `检查系统环境`、`检查 OpenClaw 配置`、`诊断 Gateway`、`一键排障` 这四张开始，基本覆盖了大部分"刚装好"和"突然坏了"的情况。
03
配置模型与启动 Gateway

配置模型，然后启动 Gateway

`模型配置` 页面决定 OpenClaw 用什么模型工作，`服务管理` 页面决定 Gateway 是否正常跑起来。这两个步骤加起来，才算把 OpenClaw 主体点亮。

  • 至少添加 1 个主模型，建议再准备 1 到 2 个备选模型。
  • 模型测试通过后保存，必要时重启 Gateway 让配置生效。
  • `服务管理` 里看 Gateway 是否绿色运行，出问题优先看这里和 `日志查看`。
过关标准： 主模型可用，Gateway 能正常启动，服务状态不是红的。
04
创建 Agent 并挂上 Skills

先创建 Agent，再给它挂上能干活的 Skills

打开 `Agent 管理`，先做出一只职责单一、名字清楚、模型明确的 Agent。然后打开 `Skills` 页面，确认当前最需要的那几个能力能不能真的调用。

  • 第一次上手建议先做单一职责 Agent，不要一开始就堆太多设定。
  • `Agent ID` 建议全小写、简单好记；`名称` 给人看，可以自然一点。
  • Skill 不只是开关，很多还依赖命令行工具、环境变量或额外配置。
  • 模型决定它聪不聪明，Skills 决定它会不会干活，身份文件决定它像不像你要的那个角色。
过关标准： Agent 列表里有一只你叫得出名字的 Agent，至少一个关键 Skill 可用。

最常见的误区

"页面里能看到 Skill，所以应该已经能用。"

更靠谱的判断方式

看它是不是可用状态，而不是只看它有没有出现在列表里。

建议

先保证 1 到 2 个关键能力真的能跑，再扩张技能树。

05
实时聊天验证

去实时聊天，让它开口

最后打开 `实时聊天`，发第一条消息。这一步最重要，因为很多配置在页面上都"像是好了"，但只有这里回得出来，才说明整条链路真的通了。

  • Gateway 在工作。
  • 模型能正常调用。
  • Agent 已经被正确加载。
  • 权限没有把它卡住。
第一次测试可以这样问
你好，请用三句话介绍你是谁、你能做什么、你现在用的是哪个模型。
过关标准： 你已经收到第一条稳定回复。到这里，才算从"安装成功"升级到"可以开工"。
06
接消息渠道

先明白规律，再接飞书和 Telegram

`消息渠道` 页面统一管理外部接入。无论接什么平台，有两个前提都一样：Gateway 要先运行，保存配置后通常都需要做平台侧的发布或配对审批。

  • 飞书 / Lark：创建企业自建应用，开启机器人能力，拿到 App ID / App Secret，事件订阅建议用长连接（WebSocket），保存后做配对审批。
  • Telegram：通过 `@BotFather` 创建 Bot，拿到 Bot Token，再准备允许的用户 ID，填到渠道配置里保存即可。
  • 保存后不生效，优先检查 Gateway 是否在运行、平台侧应用是否已发布、机器人是否已被添加到群或工作台中。
过关标准： 机器人已经能在对应平台收到并回复消息，或者你至少已经完成凭证校验和配对。

飞书 / Lark

需要 App ID、App Secret，支持内置插件和官方插件两种模式。发布应用版本后，机器人对其他人才可见。

Telegram

需要 Bot Token 和允许的用户 ID。最轻量，适合最快跑通第一条外部渠道。

一个很好用的习惯

每完成一步，如果你不确定自己现在算不算"过关"，都可以直接去问钳子助手："我刚完成了模型配置，现在下一步应该做什么？" 比自己在页面里来回猜，通常快很多。

Daily Ops

日常管理，先记住这 4 件事

ProspectClaw 用顺以后，真正省时间的不是"会配"，而是"知道平时该盯哪里"。

01

模型要稳

Agent 突然变木，大多数时候不是角色坏了，是模型掉线了。先看 API Key、Base URL、主模型连通性，再看备选模型能不能顶上。

02

Gateway 要关好门

只自己用时选"仅本机使用"最省心。要共享就配好 Token 或密码。把门打开很容易，把门关好更重要。

03

Agent 要定期整理

名字和职责写清楚，大改前先备份，一个 Agent 尽量只承担一类核心任务。这样出问题时，你知道该修谁。

04

先看日志，再找助手

排障顺序建议固定成：实时聊天表现 → 日志查看 → 系统诊断 → 钳子助手。别一出错就先重装，很多时候只是某个环节没接上。

日常高频用到的页面

`钳子助手`：问路、排障、卡片触发。`服务管理`：启动 / 停止 / 重启 Gateway，做配置备份。`日志查看`：看 Gateway 日志、错误日志、守护进程日志。`记忆文件`：按 Agent 查看工作记忆、归档和核心文件。`关于`：看推荐稳定版，做版本切换或回退。

Troubleshooting

一份够用的排障清单

不求覆盖所有奇怪情况，先把最常见的坑变成固定检查动作。

Agent 不回话

  • 先看 `实时聊天` 是否连上 Gateway。
  • 确认 `模型配置` 里的主模型测试通过。
  • 确认目标 Agent 真的绑定了模型。
  • 确认 Gateway 认证没有把请求拦住。
  • 把报错现象直接发给 `钳子助手`，让它帮你判断是模型问题、Gateway 问题还是 Agent 本身问题。

能聊天，但不会用技能

  • 看 `Skills` 页面里对应 Skill 是否真的是可用状态。
  • 检查是否缺命令行依赖、环境变量或配置。
  • 检查 Gateway 工具权限是不是设得太严。
  • 检查 Agent 规则里有没有把能力边界卡死。
  • 把 Skills 页里的状态和缺失项贴给 `钳子助手`，通常它能直接告诉你先补哪一个依赖最省事。

突然一堆报错

  • 回想最近有没有切 OpenClaw 版本。
  • 看看 `关于` 页当前版本是不是高于推荐稳定版。
  • 检查最近有没有改服务商、Token 或 Gateway 安全设置。
  • 别急着全推倒，先确认是哪个环节先变了。
  • 最省时间的做法通常是：截图或粘日志给 `钳子助手`，先让它帮你缩小排查范围。

虾养死了，怎么抢救

  • 先别慌着删 Agent、重装或全量重配。
  • 把"它之前能做什么、现在哪里不对、最近改过什么"整理成三句话。
  • 把对应报错、截图或日志丢给 `钳子助手`，直接问它："这只虾是缺氧、断粮还是网关翻车了？"
  • 然后按它给的顺序查：环境 → 模型 → Gateway → Skills → Agent 本身。

Glossary

装好以后，最好知道的 9 个基本概念

尽量说人话，不把你丢进术语海里游泳。

ProspectClaw ≠ OpenClaw

ProspectClaw 是管理面板，OpenClaw 是真正跑 Agent 的底层引擎。一个像驾驶舱，一个像发动机。

CLI 不是另一个产品

它是 OpenClaw 的命令行入口。你不一定天天手敲，但它能用，说明面板真的能接上底层。

Gateway 是总入口

聊天、模型调用、很多 Agent 行为都要经过它。很多"AI 不聪明"的问题，最后发现是入口没通。

Agent 是角色，不是聊天窗口

它有自己的名字、模型、工作区、规则、记忆和技能边界。新建 Agent，本质上是在造一个新角色。

工作区就是它的生活区

身份定义、行为规则、记忆文件、任务资料通常都在这附近。把 Agent 比成员工，工作区就是它的工位和文件夹。

Skill 是能力插件

它决定 Agent 除了聊天，还能不能调工具、连外部服务、做特定任务。有些 Skill 不是点亮就完事，还要依赖环境。

主模型和备选模型都重要

主模型是默认主力，备选模型是替补。只配一个模型当然也能跑，但一抽风就会带着 Agent 一起抽风。

钳子助手不等于你的 Agent

钳子助手更偏配置说明、排障和规则起草；实时聊天里的 Agent，才是在验证你自己养出来的那只是不是能独立工作。前者像教练兼修理工，后者才是你真正要养活的那只虾。

版本不是越新越香

推荐稳定版通常更适合长期使用。升得太快，可能先收获新特性，也可能先收获兼容性惊喜。

Invest Edition

投资管理模块（Invest 版本专属）

如果你用的是 ProspectClaw Invest 版本，以下是几个最常用的专属功能模块。

项目管线（Pipeline）

把每个项目放进对应阶段，从筛选到投后全程追踪。拖拽调整状态，历史记录完整保留。

战略评分（Scoring）

多维度评估项目战略契合度。灵活配置评分模型，系统化评估每个标的。

企业库（Companies）

集中管理潜在标的与已投企业信息，支持批量导入、自定义字段与多维度筛选。

自动化（Automation）

定时任务 + 心跳监控 + 事件触发，Agent 可以 7×24 自动运行，异常时自愈。

SOP 任务规划（Task Planner）

描述一个大目标，AI 自动拆解为可执行步骤。DAG 依赖管理、执行监督、质量门槛、流程图可视化，从规划到完成全程可追踪。

使用 EvoScientist 的前提

需要先完成基础配置（模型 + Gateway），然后在 EvoScientist 设置 Tab 配置 Provider 和 API Key，再点击工作台中的"安装 EvoScientist"按钮完成安装。安装完成后点击"启动 Bridge"即可开始使用。注意：EvoScientist 仅在 Tauri 桌面端可用，Web 模式不支持。
